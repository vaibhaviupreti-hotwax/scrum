# Product Architecture Study — May 12: The Variation Phase

## Overview: Transitioning to Variations
We have completed the **"Product Surface"** phase—documenting everything that defines a single product in isolation. We are now entering the **Variation Phase**, which handles how a single concept (e.g., a "Yoga Shirt") branches into specific sellable units (e.g., "Red/Small," "Blue/Large").

---

## Data Governance: Seed (System) vs. Customer (Business) Data
A critical architectural concept in HotWax/OFBiz is the distinction between data that the **System** owns and data that the **Business** owns.

| Data Category | Responsibility | Examples |
| :--- | :--- | :--- |
| **System-Owned (Seed)** | Handled by Developers / Framework. Defined in `ProductSeedData.xml`. | `COLOR`, `SIZE`, `FINISHED_GOOD`. |
| **Business-Owned (Customer)** | Handled by the Client / Operations team via the UI or Sync. | `WOOD_TYPE`, `VINTAGE_YEAR`, specific feature values like "Navy Blue". |

---

## Entity 17: `ProductFeatureType` — The Category Rulebook

`ProductFeatureType` defines the **Categories of Characteristics**. It does not store values like "Red" or "Small"; instead, it defines the "Bucket" that those values fall into.

### 1. The Core Logic
Before the system can understand that a shirt is "Blue," it must first know that a concept called **"Color"** exists. 
- **The "One-Liner"**: It defines the **"What"** (the kind of feature) rather than the **"Value"**.
- **The "Group" Parent**: Often includes seeded parents like `DIMENSION` to group `WIDTH`, `HEIGHT`, and `DEPTH`.

### 2. Field Breakdown: Data Provenance
| Field | Category | Technical Details |
| :--- | :---: | :--- |
| **`productFeatureTypeId`** | **Both** | Standard types (`COLOR`, `SIZE`, `BRAND`, `STYLE`) are **Seeded**. Custom types (`NECK_LINE`, `WOOD_TYPE`) are added by **Customers**. |
| **`parentTypeId`** | **Seeded** | Usually used to group standard types. |
| **`hasTable`** | **Seeded** | Always **'N'** for standard types; change requires database schema modifications. |
| **`description`** | **Both** | Seeded by default, but often **edited** by customers to match brand tone (e.g., renaming "Color" to "Shade"). |

## Entity 18: `ProductFeature` — The Global Dictionary

If `ProductFeatureType` is the **Question** ("What color?"), then `ProductFeature` is the **Answer** ("Red").

### 1. The Core Purpose: Global Reusability
The most important thing to understand is that this table is **Global**. You define "Cotton" **once** and link it to thousands of different shirts.

### 2. Field Breakdown: Data Provenance
| Field | Category | Technical Details |
| :--- | :---: | :--- |
| **`productFeatureId`** | **Customer** | 99% of this data is **Customer/Sync Added**. The system doesn't know your specific inventory until you add it. |
| **`productFeatureTypeId`** | **Customer Choice** | The user chooses which **Seeded Type** (e.g., `COLOR`) this value belongs to. |
| **`description`** | **Customer** | The human-readable name (e.g., "Navy Blue"). |
| **`uomId`** | **Seeded** | The units (e.g., `LEN_in` for inches) are **Seeded** in `UomSeedData.xml`. |

---

## Technical & Operational Insights

### 1. HotWax ID Strategy
In HotWax, `productFeatureId` is often generated by prefixing the type (e.g., `COLOR_RED`, `SIZE_SMALL`) to keep the database human-readable.

### 2. The Shopify Connection
When a product is synced from Shopify, the Shopify Connector identifies the "Options" (Size, Color) and dynamically creates a `ProductFeature` for each unique option value it encounters.

### 3. Audit & Metadata
*   **Audit Stamps**: Even for seed data, `lastUpdatedStamp` and `createdStamp` are critical. If `createdStamp` matches the system installation date, it was seeded. If it's a recent date, it was added by a user or sync.
*   **Transaction Timing**: In `ProductFeature`, the `lastUpdatedStamp` tells you exactly when a new color or size was first recognized by the system.

## Summary Reference

| Entity | Primary Source | Role |
| :--- | :--- | :--- |
| **`ProductFeatureType`** | **Seed Data** | Defines the *categories* the code depends on. |
| **`ProductFeature`** | **Customer/Sync** | Defines the *inventory values* unique to the brand. |

---

> [!NOTE]
> In an ER diagram, `ProductFeatureType` is the metadata anchor for the entire variation system. Without it, the `ProductFeature` table would be a flat list of values without any categorical meaning.
