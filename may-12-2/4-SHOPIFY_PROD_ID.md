I found the `SHOPIFY_PROD_ID` identifier and its mapping logic within the codebase. It acts as the critical bridge between Shopify's ecosystem and HotWax's internal product registry.

Here is how it is structured and mapped:

### 1. The Data Definition (Seed Data)
`SHOPIFY_PROD_ID` is officially registered as a `GoodIdentificationType` in the Shopify Connector application. 
**File**: `applications/shopify-connector/data/ShopifySeedData.xml`
```xml
<GoodIdentificationType goodIdentificationTypeId="SHOPIFY_PROD_ID" hasTable="N" description="Shopify Product ID" parentTypeId="HC_GOOD_ID_TYPE"/>

<Enumeration description="Shopify Product Id" enumCode="SHOPIFY_PROD_ID" enumId="SHOPIFY_PRODUCT_ID" sequenceId="01" enumTypeId="SHOP_PROD_IDENTITY"/>
```
*Meaning*: The database is instructed that `SHOPIFY_PROD_ID` is a valid alias category that can be attached to any Product.

### 2. The Code-Level Mapping (Java Logic)
When an order or inventory check comes in from Shopify, it brings a Shopify-specific ID (e.g., a GraphQL Variant ID). The Java service layer uses `SHOPIFY_PROD_ID` to "translate" that external ID into the internal HotWax `productId`.

**File**: `applications/shopify-connector/src/main/java/co/hotwax/shopify/product/ShopifyProductServices.java` (Inside the `checkCartInventory` method):
```java
// Sets the identifier type to look for
String identificationTypeId = "SHOPIFY_PROD_ID";

// Translates the incoming Shopify variant IDs to HotWax internal product IDs
List<String> productIds = items.stream().map(k ->
        UtilMisc.toList(NamespaceHelper.getProductId(delegator, (String) k.get("variantId"), identificationTypeId))
).flatMap(List::stream).collect(Collectors.toList());
```

### How the Mapping Works in Practice:
If Shopify says: *"Customer bought Variant ID `40912345`"*, the system does a lookup in the `GoodIdentification` table:
*   **Search**: Give me the `productId` where `goodIdentificationTypeId` = `"SHOPIFY_PROD_ID"` AND `idValue` = `"40912345"`.
*   **Result**: It returns the internal ID (e.g., `10056`), allowing the ERP to confidently deduct inventory and create the order for the correct physical item.

Yes, exactly! While they aren't technically "sub-types" in a hierarchical sense, HotWax does have **multiple specific identifiers that are exclusively used for Shopify**. 

Based on the actual codebase seed data (`ShopifySeedData.xml`), there are **two** specific `GoodIdentificationType` records dedicated purely to Shopify:

| Identifier Name (`goodIdentificationTypeId`) | Example Value (`idValue`) | Description & Purpose |
| :--- | :--- | :--- |
| **`SHOPIFY_PROD_ID`** | `gid://shopify/Variant/409` | **Shopify Internal System ID**: This is the unique, unchangeable ID that Shopify's database generates. HotWax uses this for highly accurate API syncing. |
| **`SHOPIFY_PROD_SKU`** | `MENS-TEE-RED` | **Shopify Product SKU**: This is the SKU exactly as it was typed into the Shopify admin panel. It is tracked separately because sometimes the "Shopify SKU" differs slightly from the main internal "Warehouse SKU". |

### Why do we need both?
If an ecommerce manager accidentally changes the SKU in Shopify from `MENS-TEE-RED` to `TEE-RED-M`, the sync could break if HotWax only tracked the SKU. 

By having `SHOPIFY_PROD_ID` (which never changes, even if the user edits the product in Shopify), HotWax can always securely find the right product and say: *"Ah, this is still the exact same red shirt, they just renamed the SKU on the website!"*
